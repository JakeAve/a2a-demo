Agent-to-Agent Communication

Two
primitive
shapes.

A2A agents can talk in two fundamentally different ways. Delegation is a synchronous call — you ask, you wait, you get a value back. Rooms are async conversations — you post, others reply in their own time, nobody blocks. Same system. Different topology. Different use cases.

↓ Delegation ↓ Rooms
01 — System map

Five components.
One orchestrator starts them all.

Service
Registry
:7890 · HTTP
Name → agent card directory. Agents self-register on startup with their AgentCard (name, URL, capabilities). Everyone else — other agents, the broker — resolves peer inbox URLs through here. Pure discovery: no state, no messages, no events.
Service
Monitor
:7891 · HTTP + SSE
Event sink for every observable action in the system. Agents emit turn.started, tool.call, message.completed; the broker emits all room.* events. The web UI renders everything as swimlane arrows. Fire-and-forget: monitor failure never affects agent behaviour.
Service
Room Broker
:7892 · HTTP
Single authority on room membership, the canonical message transcript, delivery records, and turn-count accounting. Serialises all mutations per room. Pushes inbox deliveries to agents. Resolves member inbox URLs via the Registry. Emits all room.* events to the Monitor.
Actor (N instances)
Agent
:dynamic · HTTP — three endpoints
GET /.well-known/agent.json — serves the AgentCard
POST /message/send — handles a synchronous delegation call (blocks until done, returns text)
POST /inbox — accepts a room delivery immediately (202), enqueues it into a serialised FIFO. The queue runs one room-turn at a time so a single Ollama instance is never overwhelmed.

Emits events to Monitor. Registers with Registry. When room-capable, holds a RoomBrokerClient in its tool deps.
Actor
REPL / Human
stdin/stdout + dynamic /inbox port
The human-facing entry point. For direct agent calls, sends POST /message/send. For rooms: starts a small Hono /inbox server (dynamic port), registers as a kind:"human" member with the broker, prints deliveries to the terminal, and turns typed lines into POST /rooms/:id/post calls. Human deliveries get a 1-hour deadline — the 120s sweep is for crashed agents.

02 — First primitive

Delegation
Synchronous RPC.

You ask. You wait. You get a value back.

Delegation is a blocking HTTP call from one agent to another. The caller posts a message to /message/send, blocks on the response, and gets a single text result. The callee sees the request, runs its full tool loop, and returns when done. The caller is frozen until that happens.

It's a synchronous RPC. A function call over the network. The shape is hierarchical: there's always a caller up the stack and a callee down it — never true peers.

  • ShapeSynchronous — caller blocks until result
  • ReturnsYes — a single text value
  • TopologyHierarchical (parent → child → grandchild…)
  • Loop safetyx-depth header, capped at max(registered agents)
  • Parties2 — exactly one caller, one callee
  • StateKV ContextStore — conversation history per contextId
  • Eventsdelegate.start / .continue / .return, turn.started / turn.completed

Strengths

  • Familiar — works like a function call
  • Returns a structured value
  • Naturally serialised — no race conditions
  • Depth cap prevents infinite loops
  • Trivial to reason about — you always know who's in control

Limitations

  • Caller is completely blocked while waiting
  • Only two parties — no group conversations
  • Depth cap limits delegation chains
  • A→B→A is forbidden (correctly — it's depth ≥ max)
  • Human can't participate in the exchange
// Agent calls another agent — blocks until result
const result = await delegate_start({
  peer: "code-reviewer",
  prompt: "Review this diff: ..."
});
// result.text is the reviewer's response

03 — Second primitive

Rooms
Async group conversation.

Post and move on. Others reply in their own time.

A room is an async, multi-party conversation channel brokered by the Room Broker. Any member can address any other member. Nobody blocks on a reply. A "turn" is just a message delivery — the recipient wakes up, processes it, and may or may not reply.

The broker is the single source of truth for membership, the canonical transcript, and delivery accounting. An agent posts to the broker; the broker pushes it to each addressed member's /inbox. The poster gets an immediate 202 and moves on — there is no return value.

  • ShapeAsync — fire-and-forget posts, no blocking
  • ReturnsNo — participation is the point, not a result
  • TopologyPeer-to-peer, N-party via a central broker
  • Loop safetyTurn count cap (maxTurns, default 24)
  • PartiesN members — agents + humans, all equal
  • StateBroker KV — transcript, deliveries, membership
  • Eventsroom.created, room.post, room.idle, room.capped, room.closed…

Strengths

  • Multi-party — any N agents + humans
  • Nobody blocks — agents stay responsive
  • Human is a first-class participant
  • Natural for debate / brainstorm / collaboration
  • Serialised per-agent — only one room turn at a time
  • Broker is the durable source of truth

Limitations

  • No return value — callers can't await a result
  • Extra hop through the broker on every message
  • Broker is a required dependency
  • More complex to observe (turn order is broker-authoritative)
  • Addressed turns only — broadcast storms avoided by design
// Create a room and start a conversation
const { roomId } = await create_room({
  title: "hotdog debate",
  members: ["analyst", "code-reviewer"]
});
// Post to specific members — no blocking
await post({ roomId, text: "Is a hotdog a sandwich?",
          to: ["analyst"] });
Room flow — async fanout
Termination conditions
  • idle No pending deliveries — a turn addressed no one. Room stays open.
  • leave A member calls leave(). Auto-closes when <2 active members remain.
  • capped turnCount ≥ maxTurns (default 24). Broker rejects further posts. Room stays open.

05 — The rule

One question
to choose between them.

"Do you need a value back?"
The callee's answer will be used directly by the caller — as a summary, a decision, a processed result. The caller can't continue until it has that value. You need the classic request / response shape.
summarize this doc review this diff search the web for X translate this text extract structured data
or
"Is the exchange itself the point?"
No single answer is expected back. You want agents to argue, explore, build on each other's ideas, or include a human in the loop. The conversation produces value — not any one turn's output.

◈
The system prompt guidance is: "need a value back → delegate_*"  ·  "want an open-ended exchange → open a room". Both primitives are always available to tool-capable agents. They are complementary — not competing — and you'll often see both in a single session: a coordinator delegates to a researcher, then opens a room with the analyst and a human to discuss the findings.
