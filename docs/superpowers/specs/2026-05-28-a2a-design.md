# A2A Prototype — Design Spec

**Date:** 2026-05-28 **Status:** Approved for implementation planning

## Goal

Build a working prototype of Agent-to-Agent (A2A) communication where a
Claude-backed agent can delegate work to one or more Ollama-backed agents, and
vice versa. The prototype must use the real A2A wire protocol (HTTP +
JSON-RPC-ish messages, Agent Cards, SSE streaming) so the lessons transfer to
production-style deployments. Built in Deno + TypeScript.

## Non-Goals

- Multi-machine / remote deployment. Localhost only.
- Full A2A v1.0 spec conformance. Implement the subset needed for round-trips.
- MCP integration. Treat MCP as a future wrapper around this system.
- Push notifications / webhooks. Skip in favor of SSE + polling.
- Persistent agent identity / signed Agent Cards. Bearer-token auth only.

## Architecture

One Deno process. Each agent binds its own OS-assigned port and registers itself
with a central registry. The orchestrator wires everything up at startup and
runs a REPL. Standalone agents can join later by pointing at the same registry.

```
deno task start --agents="sonnet,gemma3,gemma4"

  [registry]     localhost:7890          (only hardcoded port)
  [sonnet]       localhost:<dynamic>     (OS-assigned, self-registers)
  [gemma3]       localhost:<dynamic>
  [gemma4]       localhost:<dynamic>
  [REPL]         stdin / stdout

# Later, standalone:
deno task start:agent code-reviewer
  → binds port 0 → registers with localhost:7890
  → immediately callable by other agents
```

### Two layers of wrapping

1. **Orchestration layer** — `deno task start` spins up the registry, multiple
   agents, and the REPL in one process for convenience.
2. **A2A layer** — each agent is a self-contained server that knows how to
   send/receive A2A messages. An agent can run alone
   (`deno task start:agent <role>`) without the orchestrator, joining via the
   registry.

The A2A layer does not import or depend on the orchestrator. The orchestrator
imports agents as modules.

## A2A Protocol Layer

Each agent serves the same Hono routes. Auth middleware validates the shared
bearer token on every request except `/.well-known/agent.json`.

```
GET  /.well-known/agent.json   → Agent Card (public)
POST /message/send             → synchronous, returns full response
POST /message/stream           → SSE, streams tokens
GET  /tasks/:id                → poll task status
```

### Agent Card

```json
{
  "name": "gemma3",
  "description": "Fast local model for first-pass work",
  "version": "1.0.0",
  "url": "http://localhost:52341",
  "skills": [
    {
      "id": "general",
      "name": "General",
      "description": "Handles any task cheaply and quickly"
    }
  ],
  "securitySchemes": { "bearer": { "type": "http", "scheme": "bearer" } },
  "security": [{ "bearer": [] }]
}
```

### Message shape

```ts
type Part =
  | { type: "text"; text: string }
  | { type: "data"; data: unknown };

type Message = {
  messageId: string;
  role: "user" | "agent";
  parts: Part[];
  contextId?: string;
};
```

### Depth guard

Every inbound request carries an `x-depth` header. Base agent middleware:

- depth 0 → REPL/standalone caller → sonnet ✓
- depth 1 → sonnet → gemma3 ✓
- depth ≥ 2 → reject with `429 Too Many Delegations`

Each agent increments depth by 1 before forwarding to another agent via
`delegate_task`.

### Bearer token

A single shared secret in `AGENT_BEARER_TOKEN`. Agents include
`Authorization: Bearer <token>` on all calls. Wrong/missing token → `401`.
Detects accidental external exposure even if the binding is loopback-only.

## Registry

A small Hono server on port `7890` (configurable). Stores Agent Cards in memory.

```
POST   /register      body: AgentCard
DELETE /register/:name
GET    /agents        → AgentCard[]
GET    /agents/:name  → AgentCard | 404
```

Each agent on boot:

1. Binds `port: 0`, reads the OS-assigned port.
2. Constructs its Agent Card with the real URL.
3. POSTs to the registry if `REGISTRY_URL` is set; otherwise runs standalone.
4. Deregisters on `SIGINT`.

The orchestrator uses the registry too — `list_agents` in Claude's tool layer
queries `GET /agents`.

## Data Flow

```
User types: @sonnet summarize this doc, delegate the hard parts

REPL
  → POST http://localhost:<sonnet-port>/message/stream
    Authorization: Bearer <token>
    x-depth: 0
    { message: { parts: [{type:"text", text:"..."}] }, contextId: "abc" }

Sonnet handler
  → store.appendUserMessage("abc", "...")
  → store.getHistory("abc") → messages[]
  → Anthropic API streaming call with injected tools:
      list_agents()
        → GET registry/agents → AgentCard[]
      delegate_task(targetName, prompt)
        → registry lookup → target URL
        → POST <target>/message/send with x-depth: 1
          and a NEW contextId (see "Context isolation" below)
        → returns text result
  → Claude streams response; tool calls execute synchronously mid-stream
  → store.appendAssistantMessage("abc", "...")
  → SSE chunks stream back to REPL

REPL
  → prints tokens live
  → on [DONE], newline + prompt
```

## Delegation threads

A Claude-backed agent has four delegation tools and a peer-discovery tool,
giving the model explicit control over sub-conversation lifecycle:

| Tool                                    | Purpose                                                                          |
| --------------------------------------- | -------------------------------------------------------------------------------- |
| `list_agents()`                         | Discover available peers + their skills.                                         |
| `delegate_start(agent, prompt, title?)` | Start a NEW sub-conversation. Returns `{ threadId, text }`.                      |
| `delegate_continue(threadId, prompt)`   | Continue an existing sub-conversation. Peer sees prior turns.                    |
| `list_my_threads()`                     | List the parent's active threads (threadId, peer, title, turnCount, lastUsedAt). |
| `reset_thread(threadId)`                | Delete a finished sub-conversation.                                              |

### Storage layout

```
["context", <id>]                              → StoredMessage[]   (raw history)
["thread", <threadId>]                         → ThreadMeta
["thread_owner", <parentContextId>, <threadId>] → peerName          (index for list)
```

`ThreadMeta` carries: `threadId`, `peer`, `parentContextId`, `createdAt`,
`lastUsedAt`, `turnCount`, `title`.

### Flow

```
user → sonnet (contextId: A)
  store[A] += { user: "write me a haiku, ask gemma3" }
  delegate_start("gemma3", "haiku about frogs", title?)
    → threads.start(A, "gemma3", title) → threadId T
    → POST gemma3 with contextId: T
       store[T] = [user, assistant]
    → threads.touch(T)              # turnCount=1
  sonnet completes turn under contextId: A

user → sonnet (contextId: A)        # next turn
  "now make it darker"
  list_my_threads() → [{ threadId: T, peer: gemma3, ... }]
  delegate_continue(T, "darker version")
    → POST gemma3 with contextId: T
       store[T] grows; peer sees prior haiku
    → threads.touch(T)              # turnCount=2
```

### Authorization

`delegate_continue` and `reset_thread` only act if the thread's
`parentContextId` matches the caller's contextId. Cross-conversation thread
access is rejected. This protects against an agent calling on a thread that
belongs to a different parent.

### Why threads are explicit (not auto-reused)

The model decides whether a follow-up belongs in an existing thread or deserves
a fresh one. Auto-reuse ("always use the latest thread per peer") saves a tool
call but couples unrelated requests. Explicit tools cost one extra round-trip
but give the model and the user readable, debuggable delegation state.

### Why sub-context, not the parent's contextId

- The peer's system prompt and role differ from the caller's. Replaying the
  caller's history leaks coordination talk ("call delegate_task to ask
  gemma3...") into the peer's window, which produced empty responses during
  early smoke testing.
- Delegations are sub-tasks, not continuations of the parent.
- The KV stays separable per agent for debugging — `kv list ["context"]` shows
  exactly what each agent saw.

### Depth guard interaction

`delegate_start` and `delegate_continue` both increment `x-depth` by 1 before
calling the peer. Max depth remains 2 (so sonnet → gemma3 is allowed; gemma3
delegating further is rejected).

## Tool-capable Ollama agents

Any Ollama model that supports tool calling (gemma4:e4b, qwen2.5, llama3.1,
etc.) can be wired with the same A2A tool runner that Claude uses. The shared
definitions live in `src/agent/tools.ts` and are adapted at the call site to
each provider's schema:

- Anthropic format: `{ name, description, input_schema }`
- Ollama format:
  `{ type: "function", function: { name, description, parameters } }`

`RolePreset.toolCapable: true` marks a role for tool wiring. The orchestrator
constructs `ToolDeps` and passes it to `makeOllamaHandlers` or
`makeClaudeHandlers`; the loop on tool calls is identical.

**Why a flag and not auto-detect:** an Ollama model that doesn't support tools
won't error on the `tools` parameter — it will silently ignore it and produce
text. That's a confusing failure mode. Explicit opt-in via the role config keeps
surprises out of the request path.

**Tested:** gemma4:e4b autonomously calls `list_agents`, `delegate_start`,
`delegate_continue`, etc. when given a task that benefits from a peer. The
model's native `thinking` traces show it reasoning about which tool to invoke.

## Spawning agents at runtime

A Claude-backed agent in the orchestrator gets two additional tools that let it
create new peers on demand:

| Tool                               | Purpose                                                                           |
| ---------------------------------- | --------------------------------------------------------------------------------- |
| `list_roles()`                     | List role presets, each `{ name, description, backend, defaultModel }`.           |
| `spawn_agent(role, name?, model?)` | Launch a new agent as its own subprocess. Returns `{ ok, name }` once registered. |

### Flow

```
sonnet
  list_roles() → [{name: "gemma3", backend: "ollama", defaultModel: "gemma3", ...}, ...]
  spawn_agent("gemma3", "gemma-helper", "gemma3:1b")
    ↓
  Orchestrator
    Deno.Command("deno", "run", "src/agent-entry.ts",
      "--role=gemma3", "--name=gemma-helper", "--model=gemma3:1b",
      "--registry=http://localhost:7890")
    poll registry until gemma-helper appears (15s timeout)
    ↓
  spawn_agent returns { ok: true, name: "gemma-helper" }

sonnet
  delegate_start("gemma-helper", "what color is the sky?")
  ↓ peer is now reachable just like any other registered agent
```

### Subprocess model

Each spawned agent runs as a separate Deno process invoking
`src/agent-entry.ts`. It has its own KV (so its conversation history is
isolated) and registers itself with the registry URL passed in. This means:

- A spawned agent's lifecycle is owned by the orchestrator. On `SIGINT`, the
  orchestrator deregisters and `SIGTERM`s all children.
- Spawned agents could in principle outlive the orchestrator (they own their own
  ports and KV), but we always kill them at shutdown for now.
- Standalone usage works too:
  `deno task start:agent --role=gemma3
  --name=remote --registry=http://other-host:7890`
  runs without an orchestrator.

### Why spawned agents can't spawn further agents

`spawnAgent` is only wired into the orchestrator's Claude handlers. The
standalone entry point passes `undefined`, which makes the Claude backend omit
the `spawn_agent` and `list_roles` tools entirely. This keeps spawn authority in
one place (the orchestrator) and prevents runaway agent explosions. If a spawned
agent needs to invoke another peer, it uses the delegation tools, which the
depth guard already constrains.

### Naming

The `name` parameter lets the same role spin up multiple instances:
`spawn_agent("gemma3", "gemma-fast")` and
`spawn_agent("gemma3",
"gemma-careful", "gemma3:9b")` create two distinct gemma3
peers that sonnet can address independently. Names must be unique across the
registry.

## REPL

```
$ deno task start --agents="sonnet,gemma3,gemma4"
[registry]   localhost:7890
[sonnet]     localhost:52341  (claude-sonnet-4-6)
[gemma3]     localhost:47823  (gemma3)
[gemma4]     localhost:61204  (gemma4)

> @sonnet summarize this and delegate extraction to gemma4
[sonnet] working...
[sonnet] → delegate_task(gemma4, "extract figures from...")
[gemma4] working...
[gemma4] ← returned in 1.2s
[sonnet] Here is the summary: <streaming tokens>...

>
```

- `@<name>` prefix routes to the named agent.
- Live token streaming via SSE.
- Delegation events printed as structured log lines above the streaming
  response.
- `ctrl-c` cleanly deregisters all agents and exits.

## Error Handling

**Startup**

- Agent fails to bind → log, skip, continue starting others. REPL banner warns
  which agents are missing.

**Mid-task**

- Anthropic API error → SSE `{ type: "error", message }` → REPL renders in red.
  Context preserved.
- Ollama unreachable → `delegate_task` returns an error string as tool result.
  Claude sees it, decides next step.
- Max depth hit → caller receives `429`, treats it as a tool error.

**Registry**

- Registry unreachable during `list_agents` → returns `[]`, logs warning. Claude
  responds directly.

**Context**

- Deno KV writes are atomic. Failed write leaves last good state.

## Testing

### Unit

- `protocol/types.ts` validation against fixtures.
- `store/context.ts` write/append/read ordering.
- Depth guard middleware in isolation.

### Integration

- Single agent against mocked `localhost:11434`.
- `/message/send` response shape conforms to types.
- `/message/stream` SSE chunk format + `[DONE]` terminator.
- Auth: missing token → 401, wrong token → 401, correct → 200.
- Registry: register on boot, deregister on shutdown, listed correctly.

### End-to-end

- Boot registry + sonnet (mocked Anthropic) + gemma3 (mocked Ollama).
- Send task to sonnet that triggers `delegate_task`.
- Assert gemma3 received task with `x-depth: 1`.
- Assert final response includes gemma3's contribution.
- Assert KV has full conversation history.

### Manual

- Real Anthropic + real Ollama round-trip.
- Kill registry mid-session → sonnet still answers, no delegation.
- Send `x-depth: 2` → 429.
- Run `deno task start:agent code-reviewer` in second terminal → self-registers
  → callable.
