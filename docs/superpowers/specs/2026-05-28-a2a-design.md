# A2A Prototype — Design Spec

**Date:** 2026-05-28
**Status:** Approved for implementation planning

## Goal

Build a working prototype of Agent-to-Agent (A2A) communication where a Claude-backed agent can delegate work to one or more Ollama-backed agents, and vice versa. The prototype must use the real A2A wire protocol (HTTP + JSON-RPC-ish messages, Agent Cards, SSE streaming) so the lessons transfer to production-style deployments. Built in Deno + TypeScript.

## Non-Goals

- Multi-machine / remote deployment. Localhost only.
- Full A2A v1.0 spec conformance. Implement the subset needed for round-trips.
- MCP integration. Treat MCP as a future wrapper around this system.
- Push notifications / webhooks. Skip in favor of SSE + polling.
- Persistent agent identity / signed Agent Cards. Bearer-token auth only.

## Architecture

One Deno process. Each agent binds its own OS-assigned port and registers itself with a central registry. The orchestrator wires everything up at startup and runs a REPL. Standalone agents can join later by pointing at the same registry.

```
deno task start --agents="sonnet,gemma3,gemma4"

  [registry]     localhost:7890          (only hardcoded port)
  [sonnet]       localhost:<dynamic>     (OS-assigned, self-registers)
  [gemma3]       localhost:<dynamic>
  [gemma4]       localhost:<dynamic>
  [REPL]         stdin / stdout

# Later, standalone:
deno task start:agent code-reviewer
  → binds port 0 → registers with localhost:7890
  → immediately callable by other agents
```

### Two layers of wrapping

1. **Orchestration layer** — `deno task start` spins up the registry, multiple agents, and the REPL in one process for convenience.
2. **A2A layer** — each agent is a self-contained server that knows how to send/receive A2A messages. An agent can run alone (`deno task start:agent <role>`) without the orchestrator, joining via the registry.

The A2A layer does not import or depend on the orchestrator. The orchestrator imports agents as modules.

## Configuration

```
REGISTRY_PORT=7890
ANTHROPIC_API_KEY=sk-...
AGENT_BEARER_TOKEN=<shared-secret>
OLLAMA_BASE_URL=http://localhost:11434
```

Loaded via Deno's `@std/dotenv` or `deno run --env`.

## A2A Protocol Layer

Each agent serves the same Hono routes. Auth middleware validates the shared bearer token on every request except `/.well-known/agent.json`.

```
GET  /.well-known/agent.json   → Agent Card (public)
POST /message/send             → synchronous, returns full response
POST /message/stream           → SSE, streams tokens
GET  /tasks/:id                → poll task status
```

### Agent Card

```json
{
  "name": "gemma3",
  "description": "Fast local model for first-pass work",
  "version": "1.0.0",
  "url": "http://localhost:52341",
  "skills": [
    { "id": "general", "name": "General", "description": "Handles any task cheaply and quickly" }
  ],
  "securitySchemes": { "bearer": { "type": "http", "scheme": "bearer" } },
  "security": [{ "bearer": [] }]
}
```

### Message shape

```ts
type Part =
  | { type: "text"; text: string }
  | { type: "data"; data: unknown };

type Message = {
  messageId: string;
  role: "user" | "agent";
  parts: Part[];
  contextId?: string;
};
```

### Depth guard

Every inbound request carries an `x-depth` header. Base agent middleware:

- depth 0 → REPL/standalone caller → sonnet ✓
- depth 1 → sonnet → gemma3 ✓
- depth ≥ 2 → reject with `429 Too Many Delegations`

Each agent increments depth by 1 before forwarding to another agent via `delegate_task`.

### Bearer token

A single shared secret in `AGENT_BEARER_TOKEN`. Agents include `Authorization: Bearer <token>` on all calls. Wrong/missing token → `401`. Detects accidental external exposure even if the binding is loopback-only.

## Registry

A small Hono server on port `7890` (configurable). Stores Agent Cards in memory.

```
POST   /register      body: AgentCard
DELETE /register/:name
GET    /agents        → AgentCard[]
GET    /agents/:name  → AgentCard | 404
```

Each agent on boot:
1. Binds `port: 0`, reads the OS-assigned port.
2. Constructs its Agent Card with the real URL.
3. POSTs to the registry if `REGISTRY_URL` is set; otherwise runs standalone.
4. Deregisters on `SIGINT`.

The orchestrator uses the registry too — `list_agents` in Claude's tool layer queries `GET /agents`.

## Data Flow

```
User types: @sonnet summarize this doc, delegate the hard parts

REPL
  → POST http://localhost:<sonnet-port>/message/stream
    Authorization: Bearer <token>
    x-depth: 0
    { message: { parts: [{type:"text", text:"..."}] }, contextId: "abc" }

Sonnet handler
  → store.appendUserMessage("abc", "...")
  → store.getHistory("abc") → messages[]
  → Anthropic API streaming call with injected tools:
      list_agents()
        → GET registry/agents → AgentCard[]
      delegate_task(targetName, prompt, contextId)
        → registry lookup → target URL
        → POST <target>/message/send with x-depth: 1
        → returns text result
  → Claude streams response; tool calls execute synchronously mid-stream
  → store.appendAssistantMessage("abc", "...")
  → SSE chunks stream back to REPL

REPL
  → prints tokens live
  → on [DONE], newline + prompt
```

## REPL

```
$ deno task start --agents="sonnet,gemma3,gemma4"
[registry]   localhost:7890
[sonnet]     localhost:52341  (claude-sonnet-4-6)
[gemma3]     localhost:47823  (gemma3)
[gemma4]     localhost:61204  (gemma4)

> @sonnet summarize this and delegate extraction to gemma4
[sonnet] working...
[sonnet] → delegate_task(gemma4, "extract figures from...")
[gemma4] working...
[gemma4] ← returned in 1.2s
[sonnet] Here is the summary: <streaming tokens>...

>
```

- `@<name>` prefix routes to the named agent.
- Live token streaming via SSE.
- Delegation events printed as structured log lines above the streaming response.
- `ctrl-c` cleanly deregisters all agents and exits.

## Error Handling

**Startup**
- Agent fails to bind → log, skip, continue starting others. REPL banner warns which agents are missing.

**Mid-task**
- Anthropic API error → SSE `{ type: "error", message }` → REPL renders in red. Context preserved.
- Ollama unreachable → `delegate_task` returns an error string as tool result. Claude sees it, decides next step.
- Max depth hit → caller receives `429`, treats it as a tool error.

**Registry**
- Registry unreachable during `list_agents` → returns `[]`, logs warning. Claude responds directly.

**Context**
- Deno KV writes are atomic. Failed write leaves last good state.

## Testing

### Unit

- `protocol/types.ts` validation against fixtures.
- `store/context.ts` write/append/read ordering.
- Depth guard middleware in isolation.

### Integration

- Single agent against mocked `localhost:11434`.
- `/message/send` response shape conforms to types.
- `/message/stream` SSE chunk format + `[DONE]` terminator.
- Auth: missing token → 401, wrong token → 401, correct → 200.
- Registry: register on boot, deregister on shutdown, listed correctly.

### End-to-end

- Boot registry + sonnet (mocked Anthropic) + gemma3 (mocked Ollama).
- Send task to sonnet that triggers `delegate_task`.
- Assert gemma3 received task with `x-depth: 1`.
- Assert final response includes gemma3's contribution.
- Assert KV has full conversation history.

### Manual

- Real Anthropic + real Ollama round-trip.
- Kill registry mid-session → sonnet still answers, no delegation.
- Send `x-depth: 2` → 429.
- Run `deno task start:agent code-reviewer` in second terminal → self-registers → callable.

## Open Questions (Defer)

- Streaming delegation: should `delegate_task` stream tokens through Claude in real time, or wait for the full result? (V1: wait.)
- Multi-modal parts (files, images). V1 is text-only.
- Signed Agent Cards / TOFU pinning. Not in V1.
- Standalone agent CLI (`start:agent`): designed for, implemented as follow-up after round-trips work.

## Out-of-scope Reminder

If discussion drifts toward: MCP wrapping, prod deployment, multi-machine, signed identity, observability dashboards — write it down, do not build it in V1.
